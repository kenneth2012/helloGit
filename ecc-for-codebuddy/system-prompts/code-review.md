# 代码审查规则

代码审查是质量保证的关键环节。

## 审查时机

- 代码编写或修改后
- Pull Request 提交前
- 功能完成后

## 审查清单

### 1. 代码质量

- [ ] 代码清晰可读
- [ ] 命名有意义
- [ ] 函数小（<50 行）
- [ ] 文件专注（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 适当的注释

### 2. 功能正确性

- [ ] 功能符合需求
- [ ] 边界情况已处理
- [ ] 错误情况已处理
- [ ] 测试覆盖充分

### 3. 性能

- [ ] 无明显性能问题
- [ ] 数据库查询已优化
- [ ] 无 N+1 查询问题

### 4. 安全

- [ ] 无硬编码密钥
- [ ] 输入已验证
- [ ] 使用参数化查询
- [ ] 敏感数据已保护

## 审查重点

### 代码风格
- 遵循项目编码规范
- 命名一致
- 无重复代码

### 错误处理
- 所有错误已处理
- 用户友好的错误消息
- 适当的日志记录

### 测试
- 测试覆盖率 ≥ 80%
- 单元测试、集成测试、E2E 测试
- 边界情况已测试

### 文档
- 函数有文档字符串
- 复杂逻辑有注释
- README 已更新（如需要）

## 审查流程

1. 理解代码意图
2. 检查代码质量
3. 验证功能正确性
4. 检查安全
5. 验证测试
6. 提供改进建议

## 常见问题

| 问题 | 建议 |
|------|------|
| 函数过长 | 拆分为多个小函数 |
| 嵌套过深 | 使用 early return 或提取函数 |
| 重复代码 | 提取为共享函数 |
| 硬编码值 | 提取为常量或配置 |
| 缺少错误处理 | 添加 try-catch 和错误消息 |
